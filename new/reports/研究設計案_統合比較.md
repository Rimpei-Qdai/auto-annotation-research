# 自動アノテーション研究設計案の統合比較

**作成日**: 2026年3月12日  
**対象プロジェクト**: タクシー動画の運転行動自動アノテーション  
**対象バージョン**: Ver.4 以降の研究設計

---

## 1. 背景

現行システムは、Qwen2-VL 系モデルに対して複数フレーム入力と軌道描画を組み合わせることで、Ver.4 において **51.52%** の全体精度を達成している。

一方で、既存レポートから以下の課題が明確である。

- 等速走行: 高精度で認識可能
- 停止: Visual Prompting により改善
- 左折: 一定の改善あり
- 加速/減速: 0% で未解決
- 右折: 0% で未解決
- 車線変更: 0% で未解決

この結果は、現行手法が **幾何学的な進行方向の理解には強いが、定量的な時系列変化と境界事例の判別に弱い** ことを示している。

また、L2M や CoT は既存研究でも広く使われており、単独では研究上の新規性として弱い。したがって、本研究では以下 3 案を新規性候補として設計する。

- 案1: Graph-of-Driving-Behavior Reasoning
- 案2: Verifier-Guided Driving Action Inference
- 案3: Retrieval-Augmented Feedback Inference

---

## 2. 3 案の位置づけ

| 案 | 主眼 | 解決したい失敗 | 新規性の中心 |
|----|------|----------------|--------------|
| **案1: Graph Reasoning** | 行動を構成要素の関係として表現 | 右折/左折混同、車線変更と旋回の混同 | 行動分類をグラフ推論として定式化 |
| **案2: Verifier** | 初回推論を物理・交通制約で検証 | 停止/減速/等速の矛盾、右左の取り違え | 推論過程の検証と再判定 |
| **案3: RAG Feedback** | 類似事例・失敗事例を検索して補正 | 境界事例の取り違え、少数クラスの未検出 | ケースベース検索による誤り補正 |

---

## 3. 各案が現行課題にどう効くか

### 3.1 加速・減速

現状では軌道が「方向」のみを表し、速度変化が明示されていないため、加速・減速が等速走行に吸収されている。

- 案1: `speed trend`, `brake`, `trajectory spacing` をグラフノード化
- 案2: `speed≈0`, `acc_x<0`, `brake=1` などの制約検証
- 案3: 「直進だが減速」「直進だが加速」の類似事例検索

### 3.2 右折

現状では「曲がっている」までは取れても、左右の識別が弱い。

- 案1: `trajectory direction`, `blinker`, `intersection relation` を関係表現
- 案2: `right blinker + right curvature` の一貫性検証
- 案3: 右折の少数事例を直接参照して再判定

### 3.3 車線変更

現状では車線変更と左折がともに斜め軌道として見えやすく、区別が不十分。

- 案1: `lane boundary crossing` と `intersection entry` を別ノードとして管理
- 案2: `small curvature + lateral shift + no intersection` のルール検証
- 案3: 車線変更の近傍事例を検索し、旋回との違いを補足

---

## 4. 推奨順位

### 第1候補: 案1 + 案2 の統合

最も研究として強いのは、以下の組み合わせである。

- Graph Reasoning で行動を構成要素の関係として表現
- Verifier で物理・交通ルールを用いて初回推論を検証

この組み合わせは、単なる CoT ではなく、
**「構造化推論」+「整合性検証」**
として明確な主張を作りやすい。

### 第2候補: 案3 を上に重ねる

RAG は単独で主役にするより、
**Graph + Verifier の弱点を補うフィードバック機構**
として入れる方が自然である。

理由は以下の通り。

- 小規模データでの RAG 単独は新規性が弱く見えやすい
- 類似事例検索は境界事例の補正に向く
- Verifier の再判定候補生成と非常に相性が良い

---

## 5. 統合アーキテクチャ案

```text
入力動画 + センサ
    ↓
フレーム抽出 / 軌道可視化 / 車線・速度特徴抽出
    ↓
Graph Builder
    ↓
VLM 初回推論
    ↓
Verifier
    ↓
RAG で類似事例・失敗事例・判定ルール検索
    ↓
再判定 VLM
    ↓
最終ラベル
```

---

## 6. 実装優先度

### Phase 1

- Graph 用の中間特徴を定義
- Verifier の基本ルールを実装
- 既存パイプライン内で中間 JSON を出力

### Phase 2

- 速度変化付き Visual Prompting を実装
- Graph を入力に含む多段推論を実装
- 誤判定ログを保存

### Phase 3

- 過去正解データから RAG 用知識ベースを作成
- 類似事例検索を再判定ループに接続
- アブレーション実験を実施
