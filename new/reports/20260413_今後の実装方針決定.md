# 今後の実装方針決定

**作成日**: 2026年4月13日  
**対象プロジェクト**: タクシー動画の運転行動自動アノテーション  
**目的**: L2M/CoT 導入前後の比較結果を踏まえ、今後の実装方針を固定する

---

## 1. 結論

今後の実装方針は、**「Ver.4 baseline-first」** とする。

具体的には、

- まず `Ver.4` を現在の評価セットで再現し、固定 baseline として確立する
- その後の改善は、baseline を置き換えるのではなく、**baseline に対する差分モジュール**として実装する
- L2M/CoT / Graph / Verifier は、全体を複雑化する主経路ではなく、**境界事例の補正器**として使う

という方針を採用する。

一言で言えば、

**今後は「推論を足すこと」よりも、「強い Ver.4 を壊さずに、必要な補正だけを載せること」を優先する。**

---

## 2. この方針にした理由

### 2.1 Ver.4 が依然として最重要 baseline だから

既存レポート上では、L2M/CoT 導入前の `Ver.4` が

- `Qwen2-VL-2B`
- `4フレーム入力`
- `赤軌道の Visual Prompting`
- `L2M/CoT なし`

で **51.52%** を記録している。

分布差を補正しても、Ver.4 は **40%台中盤相当** の実力があった可能性が高い。  
これは、近時の L2M/Graph 系 run と同等以上である。

### 2.2 現在の複雑化は安定して上積みを作れていないから

L2M/CoT 導入後の run は、

- 41.67%
- 19.19%
- 41.41%
- 40.40%

のように不安定であり、安定した上積みとは言いにくい。

つまり今の問題は、

- reasoning が足りないこと

よりも、

- 強い visual baseline を中間要約が壊していること

にあると考える方が自然である。

### 2.3 現時点の失敗は「推論不足」より「特徴の劣化」と「再解釈の暴走」で説明できるから

これまでのログでは、以下が繰り返し問題になっている。

- `trajectory_motion_cue` の暴走
- `intersection_detected` の過信
- `speed_state` と `acceleration_cause` の矛盾
- Graph 上位候補を Level 3 が覆す
- 停止や左右折への過剰寄せ

これらは、推論の深さを増せば自然に解消する問題ではなく、

- 低品質な中間表現
- 過剰な rule / prompt / summary の重ね掛け

が主因である。

---

## 3. 採用するアーキテクチャ方針

### 3.1 主経路

主経路は、Ver.4 を基準にした **シンプルな直接分類経路** とする。

```text
4フレーム + 赤軌道描画
    ↓
VLM による直接分類
    ↓
基本ラベル出力
```

この主経路は、なるべく壊さない。

### 3.2 補助経路

L2M / Graph / Verifier は、主経路を置き換えるのではなく、**補助経路**として扱う。

```text
4フレーム + 赤軌道描画
    ↓
VLM 直接分類（主経路）
    ↓
必要なときだけ
L2M / Graph / Verifier で再判定 or 補正
    ↓
最終ラベル
```

つまり、今後の設計は

- `reasoning-first`

ではなく、

- `baseline-first, reasoning-as-corrector`

とする。

---

## 4. 今後やること

### 4.1 最優先

#### A. Ver.4 を現在の評価セットで再現する

まず最優先で、git 履歴から Ver.4 相当の構成を再現し、現在の 100 サンプル評価セットで回す。

目的:

- 本当に Ver.4 が現在版より強いか確認する
- 比較の基準点を固定する

#### B. 評価を「改善サンプル」と「悪化サンプル」に分ける

今後の比較は、単なる Accuracy だけでなく、各 sample を

- baseline で誤り → 改善版で正解
- baseline で正解 → 改善版で誤り

に分けて分析する。

これにより、「改善のつもりで壊したサンプル」を明示できる。

### 4.2 次点

#### C. L2M を常時経路から外し、条件付きで呼ぶ

L2M/Graph は常に使うのではなく、次のような曖昧ケースに限定して使う。

- 左右折と車線変更が競合している
- 停止と減速が競合している
- speed-event と turn-event が競合している
- 直接分類の信頼度が低い

つまり、**全件 reasoning** はやめ、**必要時 only reasoning** にする。

#### D. Graph は ranking 生成器に限定する

Graph の役割は、

- ラベル候補の順位付け
- 矛盾検出
- 再判定の材料提示

に限定する。

Graph が主判定器になりすぎないようにする。

#### E. Level 3 の自由生成を縮小する

今後は、自由な自然言語要約から最終ラベルを出させるよりも、

- 候補集合
- 候補順位
- 禁止条件
- 上書き可能条件

を明示した constrained selection に近づける。

---

## 5. 当面やらないこと

以下は、少なくとも baseline 比較が終わるまでは主戦略にしない。

### 5.1 ルールの継ぎ足しを続けること

停止、左右折、交差点、cue validation などに対して局所ルールを継ぎ足し続ける方針は採らない。  
理由は、run ごとの偶然に引っ張られて全体設計が見えなくなるからである。

### 5.2 Level をさらに増やすこと

Level 4、Level 5 のように reasoning 段を増やす方向は採らない。  
現状では、中間表現の劣化が主要因であり、段数増加は問題を深くする可能性が高い。

### 5.3 Graph を主経路の中心に据えること

Graph は研究上は重要だが、現時点では補助的 Graph に留める。  
baseline の再現と比較が終わる前に、Graph 中心設計へ全面移行しない。

---

## 6. 実装の優先順位

### Phase 1: baseline 固定

1. Ver.4 相当コードを復元する
2. 現在の 100 サンプルで再実行する
3. 現在版との比較レポートを作る

### Phase 2: 経路の分離

1. `direct classifier` と `reasoning corrector` をコード上で分離する
2. 直接分類だけで返すモードを正式に残す
3. 曖昧ケースのみ reasoning を呼ぶ制御を入れる

### Phase 3: 補助推論の整理

1. Graph を ranking / contradiction 専用にする
2. Level 3 を constrained selection 型へ寄せる
3. turn vs lane change、stop vs decel などの境界事例だけを focused に改善する

---

## 7. 実装ルール

今後の実装では、以下を原則とする。

### 7.1 変更単位

- baseline 主経路の変更
- reasoning 補助経路の変更

を必ず分けて行う。

### 7.2 評価単位

新しい改善は、必ず

- 全体 Accuracy
- Macro F1
- クラス別 Recall
- 改善 sample 数
- 悪化 sample 数

で評価する。

### 7.3 採用基準

以下を満たさない改善は、主経路へ入れない。

- baseline より悪化 sample が少ない
- 境界クラスで明確な改善がある
- 新しい failure mode を大きく増やさない

---

## 8. 研究としての立て方

研究としては、今後の主張を次のように整理する。

1. 強い Visual Prompting baseline（Ver.4）がある
2. 単純な L2M/CoT の複雑化は必ずしも安定改善をもたらさない
3. したがって、reasoning は主経路ではなく **boundary-case corrector** として設計し直す

この形なら、

- ベースラインの強さ
- reasoning の限界
- reasoning の再設計

が一本のストーリーとしてつながる。

---

## 9. 最終決定

今後の実装方針を以下に決定する。

- **主経路は Ver.4 baseline-first**
- **L2M/CoT / Graph / Verifier は補助経路**
- **全件 reasoning ではなく、曖昧ケースのみ reasoning**
- **まず baseline 再現と同条件比較を完了してから次の大改修へ進む**

したがって、次に着手すべき具体的な作業は、

**Ver.4 の復元と、現在の評価セットでの再実行・比較である。**
