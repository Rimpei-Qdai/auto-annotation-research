## 5. 原因分析

## 5.1 Level 1 の視覚幾何分析がほぼ崩壊している

最新 run の成功 96 件について、Level 1 出力を集計すると以下だった。

- `trajectory_relation = PARALLEL`: 96 / 96
- `intersection_detected = NO`: 96 / 96
- `direction_change = STRAIGHT`: 96 / 96
- `visual_shift = NO_SHIFT`: 96 / 96
- `road_shape = STRAIGHT`: 92 / 96
- `road_shape = CURVE_LEFT`: 4 / 96

つまり、Level 1 はほぼ全サンプルを

- 直線
- 車線平行
- 交差点なし
- 旋回なし
- シフトなし

と見なしている。

これは重大である。  
この状態では、車線変更や交差点進入を VLM が見分ける余地がほとんどない。

### 影響

- `車線変更(左)` が 0 recall
- `交差点あり/なし` に依存する左折・右折 vs 車線変更の分離ができない
- 右左折が当たっているのは、視覚的な交差点理解ではなく `gyro_z` による旋回方向の補助が主因

### 代表例

- sample 47（人手: 車線変更(左), 自動: 右折）
- sample 76（人手: 車線変更(左), 自動: 加速）
- sample 77（人手: 車線変更(左), 自動: 減速）

これらは全て Level 1 で `PARALLEL / NO / STRAIGHT / NO_SHIFT` になっている。

---

## 5.2 Level 2 の加減速推定が `DRIVER_ACCEL` に偏りすぎている

成功 96 件の Level 2 出力は以下だった。

- `acceleration_cause = DRIVER_ACCEL`: 88 / 96
- `acceleration_cause = GRAVITY`: 8 / 96
- `speed_trend = STABLE`: 96 / 96

さらに、センサ値と照合すると、

- `speed_diff < -0.5` なのに `DRIVER_ACCEL`: 31 件
- `brake = 1` なのに `DRIVER_ACCEL`: 27 件

だった。

これは、Level 2 が速度差分やブレーキ状態を正しく要約できていないことを意味する。

### 影響

- 等速走行が加速・減速へ大量誤分類
- 減速ラベルが止まらず、Graph 上で加速候補まで立つ
- Level 3 が矛盾した説明をもとに最終判断する

### 代表例

- sample 27（人手: 等速走行, 自動: 減速）
  - `speed_diff = -28.0`
  - Level 2 は `DRIVER_ACCEL + STABLE`
  - Level 3 は最終的に減速を選択
- sample 100（人手: 等速走行, 自動: 左折）
  - `speed_diff = -7.0`
  - `gyro_z = 0.239`
  - Level 2 は `DRIVER_ACCEL + STABLE`
  - Graph では `ACCELERATING` と `LEFT` が同時に立っている

---

## 5.3 Graph の速度状態ロジックが矛盾に弱い

`driving_graph.py` では、

- `acceleration_cause == DRIVER_ACCEL`

があると、`speed_diff < 0` や `brake = 1` と矛盾していても `speed_state = ACCELERATING` になり得る。

今回の run でも、

- sample 100: `speed_diff = -7.0` なのに Graph は `speed_state = ACCELERATING`
- sample 95: `brake = 1` なのに Graph 上位候補が `加速`

となっていた。

### 影響

- Graph 候補が一貫せず、Level 3 に矛盾した材料を渡してしまう
- 「左折しながら加速」「停止しながら加速」など不自然な候補構成が残る

---

## 5.4 停止ルールと人手ラベル定義のズレ

停止は recall が高い一方、precision が低い。  
これは単にモデルが弱いだけでなく、「停止」と「減速完了」「完全静止中」の定義が自動側と人手側でずれている可能性がある。

### 代表例

- sample 1（人手: 減速, 自動: 停止）
  - `speed = 0`, `brake = 1`
  - 自動側は hard rule で停止に固定
- sample 37（人手: 等速走行, 自動: 停止）
  - `speed = 0`, `brake = 0`
  - Graph 上は等速寄りだが、Level 3 は停止を選択

### 含意

- 人手アノテーションが「その瞬間の状態」ではなく「直前の行動」を見ているケースがある
- 現在の hard rule は、行動ラベルを状態ラベルに潰しすぎている

今回の run から分かる前向きな点もある。

以前の broken run と違い、今回は `0` に崩壊していない。

これは今後の改善サイクルにとって大きい。

## 7. 優先度付き改善案

## 優先度 A: まず直すべきもの

### A1. Level 1 の視覚特徴を VLM 依存から外す

現状の Level 1 は、ほぼ全件が `PARALLEL / NO / STRAIGHT / NO_SHIFT` である。  
これは推論改善以前に、入力特徴が死んでいる状態である。

改善案:

- `trajectory_relation`
  - projected trajectory の左右変位から幾何的に算出
- `turn_direction`
  - `gyro_z` と軌道曲率から算出
- `lane_crossing`
  - 軌道と基準線の交差量から算出
- `intersection_likelihood`
  - VLM 補助を残すとしても、少なくとも yes/no を丸投げしない

要点:

- 研究の本筋は推論過程の改善なので、低レベル幾何はできるだけ deterministic にする
- VLM は「意味付け」に使い、軌道の左右関係まで丸投げしない

### A2. `speed_diff` を時間正規化する

現在の `speed_diff` は、CSV 行間の単純差分であり、時間間隔で割っていない。  
そのため、

- `+40`
- `-28`
- `-59`

のような大きすぎる差分がそのまま加減速判定に入っている。

改善案:

- `delta_v / delta_t` で正規化した速度変化率に置き換える
- 可能なら clip 内の連続センサから 1～3 秒窓でトレンドを算出する
- 単純な taxi_id 単位 diff を判定根拠の主軸にしない

### A3. Graph の `speed_state` で矛盾解決を入れる

改善案:

- `speed_diff < 0` と `DRIVER_ACCEL` が矛盾したら、`ACCELERATING` を優先しない
- `brake = 1` のときは `DRIVER_ACCEL` を盲信しない
- `speed_state` は単一値ではなく、`support` と `contradiction` の両方を持つ

最低限、

- `if acceleration_cause == DRIVER_ACCEL` を先に評価する現行順序

は見直すべきである。

## 優先度 B: 次にやるべきもの

### B1. 停止と減速の定義を人手ラベルに合わせる

改善案:

- `speed = 0 and brake = 1 -> 停止` の hard rule をそのまま使わない
- 直前数秒で速度が落ちてきて停止したなら、減速を残す設計を検討する
- `停止` は「一定時間静止」条件にする

### B2. Level 3 出力を JSON 制約で厳格化する

改善案:

- `final_reasoning = "推論内容"` のようなテンプレート残りを reject
- `system / user / assistant` 混入や長文反復を invalid とみなす
- JSON parse 成功だけでなく、内容妥当性チェックを入れる

### B3. 動画終端でのフレーム抽出を適応化する

改善案:

- クリップ終端に近い場合は `num_frames=4` を固定せず、利用可能フレームに応じて再配置する
- `duration=5.0` を固定せず、残り秒数に応じて短縮する

## 8. まとめ

今回の自動アノテーション結果は、

- カバレッジ 96%
- 完了分一致率 41.67%
- Macro F1 0.226
- Cohen's kappa 0.306

であり、人手との一致はまだ十分高いとは言えない。

ただし、今回の失敗は「全部が壊れている」わけではない。  
主因はかなり明確で、次の 3 点に集約できる。

1. Level 1 が視覚幾何をほぼ抽出できていない  
2. Level 2 / Graph が加減速を誤って加速寄りに倒しやすい  
3. 停止・減速・発進の定義が人手ラベルとずれている

逆に言うと、

- Level 1 の deterministic 化
- `speed_diff` の再設計
- Graph の矛盾処理追加
- 停止ルールの再定義

を行えば、今回の主な誤分類のかなりの部分は改善余地がある。

研究的には、**「Graph を入れたのに精度が伸びない」のではなく、Graph に入る前の特徴と整合性処理がまだ不安定**という理解が最も正確である。
