## 変更詳細と理由

### 1. `config.py` — 閾値定数・few-shot例の追加

**変更内容**
```python
GYRO_THRESHOLD = 0.1        # rad/s - 旋回検出のジャイロ閾値
ACC_X_THRESHOLD = 0.3       # m/s² - 横加速度による旋回検出閾値
SPEED_STOP_THRESHOLD = 1.0  # km/h - 停止とみなす速度閾値

FEW_SHOT_EXAMPLES = """
例1: gyro_z=+0.25 rad/s、速度=15km/h、交差点で右に曲がっている → 7（右折）
例2: gyro_z=+0.18 rad/s、速度=20km/h、車線が左にシフト → 8（車線変更（左））
...
"""
```

**理由**
- 旋回検出の閾値がコード各所にハードコードされると保守が困難になるため一元管理
- few-shot例を定数化することでプロンプト変更時の管理を容易にする

### 2. `heron_l2m_pipeline.py` — L2Mパイプライン強化

#### 2-1. Level 1（幾何学的分析）にステップ4を追加（案1-1）

**変更内容**

プロンプトに以下の質問を追加し、JSON出力に3フィールドを追加した。

```
[ステップ4: 方向転換の兆候（重要）]
(a) 交差点または角が前方に見えるか → YES / NO
(b) 車両の進行方向が変化しているか → TURNING / STRAIGHT
(c) 映像が左右にシフトしているか   → SHIFT_LEFT / SHIFT_RIGHT / NO_SHIFT
```

出力JSONに追加されたフィールド:
```json
{
  "intersection_detected": "YES",
  "direction_change": "TURNING",
  "visual_shift": "SHIFT_RIGHT"
}
```

**理由**

従来のLevel 1は道路形状・軌道関係のみを分析していた。交差点進入中でも「道路がカーブしていない + 軌道がPARALLEL」と判定され、旋回行動が直進として処理されることがあった。交差点・映像シフトを明示的に問うことで、旋回の視覚的証拠をLevel 2・3に引き継げるようにした。

---

#### 2-2. Level 2（物理整合性）にgyro_z・ウィンカーと警告を追加（案1-2）

**変更内容**

センサーデータの提示にジャイロとウィンカーを追加:
```
- ジャイロ（旋回速度）: gyro_z rad/s（正=左旋回、負=右旋回）
- ウィンカー: 右ウィンカーON / 左ウィンカーON / OFF
```

閾値超過時に警告文をプロンプトに動的挿入:
```python
if abs(gyro_z) > GYRO_THRESHOLD:
    # ⚠️ ジャイロセンサーが旋回を検出しています（gyro_z=0.xxx rad/s）。右への旋回を考慮してください。

if abs(acc_x) > ACC_X_THRESHOLD:
    # ⚠️ 横加速度あり（acc_x=0.xxx m/s²）。旋回・車線変更の可能性があります。
```

**理由**

従来のLevel 2は速度・加速度・ブレーキのみを参照していた。旋回の最も直接的な証拠であるジャイロ値が無視されていたため、旋回中でも「STABLE（安定走行）」と判定されていた。閾値超過時の警告文により、VLMが旋回の可能性を見落とすことを防ぐ。

---

#### 2-3. Level 3（最終分類）プロンプトの全面強化（案1-3・案5-2）

**変更内容**

① 全11クラスを明示し、等速走行(1)を最後の手段として位置づけ:
```
【必ず以下の11クラスから1つ選択してください】
...（全クラス列挙）...

【等速走行(1)は最後の手段】
旋回(6,7,8,9,10) → 速度変化(2,3) → 停止/発進(4,5) → 最後に等速走行(1)
```

② few-shot例（`FEW_SHOT_EXAMPLES`）を冒頭に提示

③ センサーヒント・speed_diff・ウィンカーをプロンプトに追加

**理由**

「Output the number 0-10」という従来の指示では、モデルが学習済みの最頻出クラスに収束しやすい。全クラスを明示して選択順序を指定することで、少数クラスへの注意を強制的に向けさせる。few-shot例は旋回クラスの判断パターンを想起させるため有効。

---

#### 2-4. `_get_sensor_hints()` メソッド追加（案2-1）

**変更内容**

センサー値からVLMへのヒント文字列を生成する新規メソッド:

```python
def _get_sensor_hints(self, sensor_data):
    # 停止の可能性 → "センサー: 速度≈0かつブレーキON → 停止(4)の可能性が高い"
    # 旋回検出    → "センサー: 旋回検出 (gyro_z=0.25) → 左折・車線変更の可能性"
    # 横加速度    → "センサー: 横加速度あり → 旋回の可能性"
    # ウィンカー  → "センサー: 右ウィンカーON → 右折(7)または車線変更右(9)の可能性"
```

生成されたヒントはLevel 3プロンプトの「センサーヒント」セクションに追加される。

**理由**

VLMは映像から旋回を視覚的に確認できない場合がある（軌道描画の精度限界、カメラ角度など）。センサーから明示的に「これは旋回と思われる」というヒントを与えることで、映像判断を補強できる。

---

#### 2-5. `_post_process_label()` メソッド追加（案4）

**変更内容**

VLMの予測がセンサーと著しく矛盾する場合にルールベースで補正する新規メソッド:

```python
# ルール1: 速度≈0かつブレーキON → 停止(4)に強制補正
if speed < 1.0 and brake > 0 and predicted_label != 4:
    return 4

# ルール2: gyro_z > 0.15 なのに等速走行(1) → 矛盾を警告ログ出力
if abs(gyro_z) > GYRO_THRESHOLD * 1.5 and predicted_label == 1:
    return None  # 呼び出し側で元の予測を維持
```

**理由**

L2M推論でも誤分類が発生し得る。センサーが物理的に矛盾する状況（完全停止中なのに等速走行と予測）を検出してルールで修正することで、明らかな誤りを防ぐ安全網として機能する。

### 3. `app.py` — 速度差分の事前計算（案2-2）

**変更内容**

CSVロード後に `speed_diff` を計算:
```python
df_auto = df_auto.sort_values(['taxi_id', 'timestamp']).reset_index(drop=True)
df_auto['speed_diff'] = df_auto.groupby('taxi_id')['speed'].diff().fillna(0)
```

sensor_dataに追加（3箇所すべて対応）:
```python
sensor_data = {
    ...
    'speed_diff': float(row['speed_diff']) if 'speed_diff' in row.index else 0.0
}
```

**理由**

瞬間速度だけでは「今まさに加速中か減速中か」が判断しにくい。前サンプルとの差分（speed_diff > 0 なら加速傾向、< 0 なら減速傾向）をVLMに提示することで、加速(2)・減速(3)の検出精度向上が期待できる。
