# 2026-04-20 v5ベースライン再起点 改善案と実装計画

## 1. 目的

本ドキュメントは、`sensor_video_late_fusion_v5_direct_macro4_with_trajectory` を運用ベースラインとして固定したうえで、

- これ以上どこを小手先に触っても伸びにくいのか
- 次に何を作り直すべきか
- どの順で実装・検証するべきか

を整理するための実装計画メモである。

---

## 2. 結論

結論は明確である。

1. 次にやるべきことは、`v5` の閾値調整や prompt 延命ではない
2. 根本課題は、現在の `sensor-temporal` が **dense telemetry 時系列**ではなく、[`annotation_samples.csv`](/Users/rimpeihata/Desktop/auto-annotation-research/new/sample/annotation_samples.csv) の近傍行を拾った擬似 temporal であること
3. 77% を再現性ある形で超える本命は、**dense temporal sensor を主判定器にした階層モデル**を作り、VLM を不確実例だけの specialist に下げること
4. fusion は fixed bonus ではなく、**sample-wise reliability を見る learned calibrator** に置き換えるべきである
5. graph は最終ラベルを積極的に動かす役ではなく、**verifier / contradiction checker** に格下げするべきである

要するに、次の勝ち筋は

- `sensor-first`
- `dense temporal`
- `hierarchical`
- `specialist VLM`
- `learned fusion`

であり、`v5` の微修正ではない。

## 4. 根本原因

## 4.1 現在の temporal は dense telemetry ではない

現実装の `_build_sensor_temporal_context()` は、[`annotation_samples.csv`](/Users/rimpeihata/Desktop/auto-annotation-research/new/sample/annotation_samples.csv) を temporal index として使い、同一 taxi の近傍 sample から窓特徴を作っている。

参照:

- [heron_model_with_trajectory.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/heron_model_with_trajectory.py:209)
- [heron_model_with_trajectory.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/heron_model_with_trajectory.py:268)

これは「元の連続 probe/telemetry」ではなく、「100件の評価用 sample CSV を時系列っぽく再利用している」状態である。  
したがって、ここでいくら guard や route を工夫しても、sensor-first temporal の本命にはならない。

## 4.2 `v5` の sensor は hand-crafted rule に依存している

`_build_sensor_direct_macro_scores()` は

- `A` に強い prior を加える
- 停止/発進/減速も `A` に寄せる
- turn は gyro / heading / blinker を rule で加点する

という構造である。

参照:

- [heron_model_with_trajectory.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/heron_model_with_trajectory.py:1420)

この設計は `v5` の branch 定義には合っていたが、fair 指標まで含めて安定超過を狙うには限界がある。

## 4.3 fusion が固定ボーナスである

現在の macro fusion は

- video が選んだクラスに `VIDEO_CHOICE_BONUS = 0.35` を足す

という固定 late fusion である。

参照:

- [config.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/config.py:41)
- [heron_model_with_trajectory.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/heron_model_with_trajectory.py:1577)

sample ごとの

- sensor の信頼度
- temporal quality
- video の信頼度
- graph の矛盾度

を見ていないため、これ以上の安定改善は難しい。

## 4.4 graph が verifier ではなく反転器として残っている

macro graph は `veto-only` と呼んでいるが、実際には最終ラベルをかなり動かす。

参照:

- [driving_graph.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/driving_graph.py:387)
- [heron_model_with_trajectory.py](/Users/rimpeihata/Desktop/auto-annotation-research/new/annotation_tool/heron_model_with_trajectory.py:1662)

`v5` のときは net positive だったが、後続 run では `A -> D` や `A -> B/C` の harm も作った。  
つまり graph を強くする方向は、再び閾値調整地獄に戻る危険が大きい。

## 5. 改善案

## 5.1 ベースライン固定

runtime の比較基準は `v5 direct` に固定する。

- prompt version: `sensor_video_late_fusion_v5_direct_macro4_with_trajectory`
- macro path: raw video + normalized summary 1枚
- fusion: direct sensor macro prior + fixed video bonus + veto-only graph

この baseline は今後の比較基準であり、以後はここに小さな修正を積むのではなく、**別系統の研究実装**を比較する。

## 5.2 dense telemetry dataset 層を新設

最優先で追加すべきなのは、評価 sample ごとに

- `center timestamp`
- `t-3s ~ t+3s` の連続 sensor sequence
- gap / missing / source quality

を保存する研究用 dataset builder である。

ここでは少なくとも

- `speed`
- `acc_x`
- `gyro_z`
- `heading`
- `brake`
- `blinker_l`
- `blinker_r`

を時系列で取れるようにする。

**この層がない限り、sensor-first temporal 学習は始められない。**

## 5.3 主判定器を learned sensor hierarchy にする

構造は次の3段を基本とする。

1. `STOPLIKE vs NON-STOPLIKE`
2. `TURNLIKE vs STRAIGHTLIKE`
3. `LEFT vs RIGHT`

最初のモデルは deep でなくてよい。

- Logistic Regression
- LightGBM
- XGBoost

のいずれかで、

- grouped CV (`taxi_id` 単位)
- OOF 予測の保存
- calibration

を優先する。

## 5.4 VLM は specialist verifier に下げる

VLM は全件 macro 分類をやらせない。

役割は次に限定する。

- `TURNLIKE` だが `LEFT/RIGHT` の margin が小さいときだけ `B/C specialist`
- `STOPLIKE` と `STRAIGHTLIKE` の margin が小さいときだけ `A/D specialist`

また、入力は raw clip だけでなく

- top-down summary
- normalized summary
- speed / brake / gyro timeline

を使った motion-aware 入力にする。

## 5.5 fusion を learned calibrator にする

最終 macro は hand-tuned bonus ではなく、OOF 予測を使った小さい calibrator で決める。

入力候補:

- sensor posterior
- sensor margin
- temporal source / gap / quality
- VLM specialist answer
- VLM self-consistency
- graph contradiction score

graph はここでは「ラベルを変える器」ではなく、特徴量供給側に回す。

## 5.6 graph を verifier に格下げする

graph の役割は次に限定する。

- 明白な矛盾を `needs_review` にする
- confidence を下げる
- 説明を残す

最終ラベルを積極的に反転させる役は、calibrator へ移す。

## 5.7 weak supervision を準備する

もし `kiwi` 上に未ラベル raw log が十分あるなら、

- sensor rule
- graph verifier
- VLM specialist

を labeling function として使い、probabilistic label を作る。

そのうえで disagreement slice を優先的に人手追加する。

## 7. 評価方針

毎 run で必ず出す指標は次とする。

- 11分類 exact
- legacy 4分類 Accuracy / Macro F1 / Balanced Accuracy
- report-compatible 4分類 Accuracy / Macro F1 / Balanced Accuracy
- temporal source 別成績
- route / specialist 使用率
- graph rescue / harm
- slice 別成績

重点 slice:

- `snapshot_only`
- `dense temporal available`
- `A -> B/C`
- `A -> D`
- `D -> A`
- `missing video`

---

## 8. やらない方がよいこと

- `v5` の turn 閾値だけを再調整する
- prompt をさらに細かくする
- graph veto をさらに強くする
- legacy 77.78% だけを目標に最適化する

これらは一時的に数値を押し上げる可能性はあるが、再現性のある改善にはつながりにくい。
