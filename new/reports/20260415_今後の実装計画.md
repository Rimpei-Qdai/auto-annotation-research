# 今後の実装計画

**作成日**: 2026年4月15日  
**対象プロジェクト**: タクシー動画の運転行動自動アノテーション  
**目的**: Ver.4 以前・L2M/CoT・Graph の分析結果を踏まえ、今後の実装を段階的に整理する

---

## 1. 背景

これまでの分析から、以下が明らかになった。

- L2M/CoT 導入前の **Ver.4** は、
  - `4フレーム`
  - `赤軌道 Visual Prompting`
  - `直接分類`
  というシンプルな構成で強い baseline だった
- L2M/CoT + Graph 系は、研究上は有意義だったが、
  - 中間表現の崩れ
  - センサー整合性の不足
  - Level 3 の再解釈の暴走
  により、安定した上積みを作れなかった
- したがって今後は、
  - **Ver.4 を主経路**
  - **reasoning は補助経路**
  とするべきである

---

## 2. 実装方針の要約

今後の実装方針は次の一文に集約される。

**Ver.4 baseline-first で主経路を再構築し、L2M/CoT / Graph / Verifier は曖昧ケースだけに使う補助機構として再設計する。**

---

## 3. 全体アーキテクチャ方針

### 3.1 主経路

```text
4フレーム + 赤軌道描画
    ↓
VLM による直接分類
    ↓
主予測ラベル
```

この主経路は、なるべく単純に保つ。

### 3.2 補助経路

```text
4フレーム + 赤軌道描画
    ↓
VLM による直接分類
    ↓
曖昧ケース判定
    ↓
必要なときだけ
deterministic evidence / Graph / Verifier
    ↓
再ランキングまたは補正
    ↓
最終ラベル
```

つまり今後は、

- **全件 reasoning**

ではなく、

- **必要時のみ reasoning**

に切り替える。

---

## 4. 実装の基本原則

### 4.1 baseline を壊さない

- Ver.4 の主経路は常に比較対象として残す
- 新しい改善は、baseline を置き換えるのではなく差分として実装する

### 4.2 中間特徴は「決定」ではなく「evidence」

今後使う deterministic feature は、途中で

- `PARALLEL`
- `STRAIGHT`
- `NO`

のような粗い確定ラベルに早く落としすぎず、

- `trajectory_curvature`
- `lateral_offset`
- `trajectory_point_like`
- `visible_length`
- `speed_change_rate`
- `blinker_consistency`
- `intersection_prior`

のような **判断材料** として保持する。

### 4.3 Graph は主判定器にしない

Graph の役割は以下に限定する。

- evidence の整理
- 候補順位付け
- contradiction の検出
- 再判定時の材料提示

### 4.4 Level 3 は自由生成を減らす

最終段は長い自然言語 reasoning に依存せず、

- 候補集合
- 候補順位
- 禁止条件
- 上書き条件

を与えた constrained selection に近づける。

### 4.5 改善は sample 単位で評価する

新しい実装は毎回、

- baseline で正解 → 改善版で誤り
- baseline で誤り → 改善版で正解

を必ず出す。

## 6. 実装 Phase

## Phase 1: Ver.4 baseline の再現

### 目的

- 現在の 100 サンプル評価セットで Ver.4 を再現する
- baseline を固定する

### やること

1. `main` から新ブランチを切る
2. Ver.4 相当の経路を再構築する
   - `4フレーム`
   - `赤軌道`
   - `直接分類`
   - `L2M/CoT なし`
3. 現在の評価セットで再実行する
4. 最新版との比較レポートを出す

### 対象ファイル

- `new/annotation_tool/heron_model_with_trajectory.py`
- `new/annotation_tool/visual_prompting.py`
- 必要なら `new/annotation_tool/config.py`

### 完了条件

- Ver.4 相当 run が再現できる
- 100 サンプルの結果 CSV と比較レポートが出る

---

## Phase 2: direct classifier と reasoning corrector の分離

### 目的

- 主経路と補助経路をコード上で分離する

### やること

1. 直接分類専用の推論経路を正式に残す
2. `reasoning_corrector` の入口を分離する
3. 補助推論を呼ぶ条件を定義する

### 曖昧ケースの候補条件

- top-1 と top-2 の差が小さい
- 左右折と車線変更が競合している
- 停止と減速が競合している
- 等速と加減速が競合している

### 対象ファイル

- `new/annotation_tool/heron_model_with_trajectory.py`
- `new/annotation_tool/heron_l2m_pipeline.py`

### 完了条件

- 直接分類 only モード
- 補助推論ありモード

の 2 経路を切り替え可能にする

---

## Phase 3: deterministic evidence の整理

### 目的

- 赤軌道とセンサーから取れる feature を、補助 evidence として安定化する

### やること

1. 以下の feature を正式に定義する
   - `trajectory_curvature`
   - `lateral_offset`
   - `trajectory_point_like`
   - `visible_length`
   - `speed_change_rate`
   - `blinker_consistency`
   - `intersection_prior`
2. それらを途中で確定ラベル化せず、evidence として保持する
3. evidence から候補クラスへの支持度を定義する

### 対象ファイル

- `new/annotation_tool/visual_prompting.py`
- `new/annotation_tool/heron_model_with_trajectory.py`
- `new/annotation_tool/driving_graph.py`

### 完了条件

- feature の意味と使い方がコードとレポートで一致している
- `PARALLEL / NO / STRAIGHT` のような早すぎる離散化への依存が減っている

---

## Phase 4: Graph の役割を再設計

### 目的

- Graph を説明補助・候補整理に限定し、主判定器化を防ぐ

### やること

1. Graph の入力を evidence ベースにする
2. Graph の出力を以下に限定する
   - candidate ranking
   - support
   - contradiction
3. Graph が直接 final label を決めないようにする

### 対象ファイル

- `new/annotation_tool/driving_graph.py`
- `new/annotation_tool/heron_l2m_pipeline.py`

### 完了条件

- Graph の役割が「候補整理」と「矛盾検出」に明確化される

---

## Phase 5: Verifier の限定導入

### 目的

- 境界事例だけを補正する verifier を導入する

### やること

1. 以下の競合ペアに限定して verifier を作る
   - 停止 vs 減速
   - 左折 / 右折 vs 車線変更
   - 等速 vs 加速 / 減速
2. verifier は evidence と候補のみを見る
3. verifier が再判定した理由を短く残す

### 対象ファイル

- `new/annotation_tool/heron_l2m_pipeline.py`
- 必要なら `new/annotation_tool/verifier.py` を新設

### 完了条件

- 全件 verifier ではなく、曖昧ケース限定で動作する

---

## 7. 実装しないこと

今後しばらくは、以下を主戦略にしない。

- 全件 L2M/CoT
- Level をさらに増やす
- 自由生成の長い reasoning を増やす
- ルールの継ぎ足しだけで精度を上げようとする
- Graph を主経路の中心に据える

---

## 8. 評価計画

毎フェーズで最低限以下を比較する。

- Accuracy
- Macro F1
- クラス別 Recall
- 改善 sample 数
- 悪化 sample 数

特に注目するクラス:

- `停止`
- `減速`
- `左折`
- `右折`
- `車線変更(左)`
- `車線変更(右)`
- `等速走行`
- `加速`

---

## 9. 当面の具体的タスク

直近の着手順は以下とする。

1. `main` から Ver.4 baseline 用ブランチを切る
2. Ver.4 相当構成を復元する
3. 現行 100 サンプルで再実行する
4. baseline vs 現在版の比較レポートを作る
5. その結果を見て、曖昧ケース判定条件を定義する
6. その後に only-if-needed の補助推論を実装する

---

## 10. まとめ

今後の実装は、

- **Ver.4 を主経路に戻す**
- **deterministic feature は evidence として使う**
- **L2M/CoT / Graph / Verifier は補助に限定する**
- **全件 reasoning をやめる**

という方向で進める。

一言で言えば、

**「まず見て分かるようにし、それでも迷うときだけ考える」構造へ戻すのが、今後の実装方針である。**
