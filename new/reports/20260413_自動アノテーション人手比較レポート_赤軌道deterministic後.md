## 1. まず結論

今回の run では、ローカルへ反映した自動アノテーション結果と人手アノテーション結果を比較したところ、以下の結果だった。

- 予測成功: 99 / 100 件
- 一致数: 39 / 99 件
- Accuracy（予測成功分）: 39.39%
- Accuracy（全100件基準）: 39.00%
- Macro F1: 0.197
- Cohen's kappa: 0.233
- 未出力: sample `68` の 1 件

精度そのものはまだ十分ではないが、今回の修正で **「赤い予測軌道の幾何を deterministic に読む」層は機能している**。一方で、最終精度を押し上げ切れていない主因は以下の 4 点である。

1. `intersection_detected` が 99/99 件で `NO` のままで、左折/右折と車線変更の分離に必要な文脈が入っていない。
2. `motion_feature_reliable=False` が 71/99 件で、速度変化の証拠がまだ弱い。
3. 停止判定が人手定義より狭く、`停止 -> 等速走行` の取りこぼしが目立つ。
4. Level 3 が Graph 上位候補より自然言語の印象を優先してしまい、車線変更候補を左折/右折へ戻す例が残っている。

### 5.1 何が起きているか

- `停止 -> 等速走行`
  - 赤軌道が `POINT` であっても stop likelihood が十分上がり切らないケースが残る。
  - 人手は「ほぼ停止」や「停止直前」も停止に含めている可能性が高い。
- `等速走行 -> 左折 / 右折 / 車線変更(右)`
  - 赤線の横偏位や曲率が小さくても、3秒先予測が少しでも曲がると旋回系候補に寄りやすい。
  - 特に `intersection_detected=NO` が固定なので、直進コンテキストが弱い。
- `車線変更(左) -> 左折`
  - Graph では車線変更が上位でも、Level 3 が自然言語の説明で左折側へ戻している。

## 7. ログから見えた「まだダメな点」

### 7.1 交差点コンテキストが完全に死んでいる

- `intersection_detected=NO`: 99 / 99

このため、左折/右折と車線変更を分けるための最重要文脈が欠けている。  
いくら赤軌道が曲がっても、それが「交差点での曲がり」なのか「車線変更」なのかの判断が弱い。

### 7.2 速度変化の信頼できる証拠が少ない

- `motion_feature_reliable=False`: 71 / 99
- `speed_trend=STABLE`: 65 / 99

`timestamp_diff_sec` が大きすぎるサンプルでは速度差分を無効化しているため、以前より安全になった一方で、今度は「十分な速度証拠がない」状態が多い。  
その結果、停止と減速、等速と減速、等速と加速の分離がまだ弱い。

### 7.3 Level 3 が Graph 上位候補を守り切れていない

ログ末尾の sample `100` では、

- Graph 1位: `8 車線変更(左)` score=0.75
- Graph 2位: `6 左折` score=0.60

であるにもかかわらず、Level 3 は `6 左折` を返している。  
つまり、Graph を入れても、最終段が自由に上書きできる構造が残っている。

### 7.4 車線変更(右)の過剰予測

人手では `車線変更(右)` は 0 件だが、自動では 7 件出ている。  
右方向への横偏位や `CROSSING_RIGHT` を、そのまま `車線変更(右)` に変換しすぎている可能性が高い。

## 9. 原因の整理

今回の精度が伸び切らなかった原因は、優先度順に並べると次の通りである。

### 原因 A: 交差点情報がない

左右折と車線変更の分離では、`intersection_state` が最重要だが、現在は全件 `NO` で固定されている。  
赤軌道だけで方向は読めても、「どこで曲がるか」は読めない。

### 原因 B: 停止/発進の定義が人手に合っていない

人手アノテーションでは、完全停止だけでなく停止遷移に近い状態も `停止` と見なしている可能性がある。  
現状は point-like な赤線や低速状態を使っているが、判定境界がまだ厳しすぎる。

### 原因 C: Graph と Level 3 の役割分担が曖昧

Graph が候補順位を出しても、Level 3 が自然文の印象で別ラベルに戻せるため、構造化推論の効果が弱まっている。

### 原因 D: 速度変化は安全になったが、まだ弱い

`speed_change_rate` をむやみに信じない改善は正しかったが、その代わりに信頼できる加減速証拠が減り、`STABLE` 偏重が残っている。

## 10. 優先度付き改善案

## 優先度 A: 次にまず直すべきもの

### A1. `intersection_detected` を VLM 丸投げから外す

現状の最大ボトルネックである。  
少なくとも以下を deterministic または verifier で入れるべきである。

- 道路の開口部や分岐形状の検出
- 軌道の消失先が道路中央か脇道か
- 連続フレームでの車線境界の発散

最初の実装としては、VLM の yes/no をそのまま使わず、以下の 3 値にするのがよい。

- `YES`
- `NO`
- `UNCERTAIN`

Graph では `UNCERTAIN` を「車線変更を弱く支持、左右折も弱く支持」に留め、強い加点をしないようにする。

### A2. Level 3 の自由度を下げ、Graph 上位候補を守る

現状は Graph が 1 位でも Level 3 が別候補に戻せる。  
以下の制約が必要である。

- Graph 1 位と 2 位の差が十分大きいときは、Level 3 は上位 2 候補以外を選べないようにする
- `Graph 1位 = 車線変更` かつ `intersection_state = NO` のときは、左右折への上書きを禁止する
- Level 3 には自由記述よりも、候補選択型 JSON を返させる

### A3. 停止/発進の定義を人手ラベルに合わせる

今回最大の誤りは `停止 -> 等速走行` である。  
改善案は以下。

- `trajectory_point_like=True` かつ `speed <= 12 km/h` なら stop likelihood を強く上げる
- 1フレーム瞬間値ではなく、clip 内での「低速継続」を停止証拠にする
- `発進` は `直前 stop-like + 現在 speed increase` の複合条件で定義する

## 優先度 B: その次に直すもの

### B1. 速度トレンドを clip ローカルに作り直す

今は `motion_feature_reliable=False` が 71/99 件ある。  
サンプル行間差分ではなく、動画に対応した短窓センサ系列から

- 平均速度
- 傾き
- 最終 1 秒の変化量

を直接作るべきである。

### B2. 右車線変更の過剰予測に verifier を入れる

`CROSSING_RIGHT` だけで `車線変更(右)` に寄るのを防ぐため、

- heading change が小さい
- 横偏位が一定以上
- 交差点文脈なし

の 3 条件を満たしたときだけ強く加点する。  
それ以外は `RIGHT_SHIFT` 程度の弱い証拠に落とすべきである。

## 11. まとめ

今回の run では、**赤い予測軌道の曲がり具合と長さを deterministic に読む改善そのものは効いている**。  
その結果、Level 1 の幾何特徴は以前よりかなり生き返った。

ただし、精度を押し上げるうえで本当に足りていないのは、

- 交差点という文脈
- 停止/発進の人手定義との整合
- Graph を最終判断にきちんと効かせる制約

の 3 点である。

次の実装優先順位は、`intersection_state` の改善、`Level 3` の Graph 拘束、`停止/発進` 定義の見直し、の順が妥当である。
