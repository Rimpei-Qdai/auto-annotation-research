## 2. 何が起きたか

今回の run では、以前の `S/A` 一極化は崩れたが、今度は `B=左回転系` に偏った。
特に重要なのは、`stage_traces` を見ると以下だった点である。

- Stage1 deterministic: `S=37, R=33, O=26`
- Stage2 rotation: `L=33, R=0`

つまり、回転系に入った 33件は、すべて Stage2 で `L` と判定されていた。
これは parser の問題ではなく、`generated_text` 自体が毎回 `assistant -> L` だった。

## 3. 原因整理

### 3.1 Stage2 の左右判定を VLM に任せると `L` に崩壊した

Stage2 prompt は `L / R` の二択だったが、実際には 33件すべてで `L` が返っていた。
ログ上は、`gyro_z < 0` や `trajectory_features.end_lateral_offset_m < 0` の右回転寄りサンプルでも `L` が返っていた。

これは、VLM が trajectory summary の左右符号を安定して grounding できていないか、
二択 prompt の先頭候補へ強く寄っている可能性が高い。

### 3.2 Stage1 の turn 判定がやや広すぎた

Stage1 は `R` を 33件出したが、その中には高速度・小曲率・小gyroで、
人手では直線系や減速寄りのサンプルも含まれていた。

典型例:
- sample 3: `forward_distance=19.9m`, `lateral_offset=1.46m`, `heading_delta=0.146`, `gyro=0.05`
- sample 23: `forward_distance=39.0m`, `lateral_offset=2.63m`, `heading_delta=0.134`, `gyro=0.046`

これらは絶対横偏位だけ見ると turn に見えるが、前進距離に対する比率で見ると弱い。

### 3.3 現状の VLM は「軌道形状の符号」より「回転っぽい summary」を雑に読む

今回の summary 分離で、raw frame と軌道図を分ける改善自体は有効だった。
ただし、`左か右か` のような符号付き幾何は、まだ VLM に直接読ませるには不安定だった。

## 4. 改善方針

今回の run を踏まえると、より抜本的には以下が必要である。

1. `R/S/O` だけでなく `L/R` も deterministic 幾何で先に切る
2. Stage1 は絶対横偏位ではなく、`lateral_ratio = lateral_offset / forward_distance` を使って保守化する
3. VLM は「回転かどうか」「左か右か」の決定器ではなく、曖昧ケースの補助に落とす
4. 将来的には VLM の役割を、turn vs lane-change の fine distinction に限定する

## 5. 今回実装した修正

このレポート作成時点で、次の修正をすでにローカル実装した。

- Stage1 に `lateral_ratio` を導入し、turn 判定を保守化
- Stage2 に deterministic な左右判定を追加
  - signed lateral offset
  - signed heading delta
  - signed gyro_z
  から `left_score / right_score` を構成して方向を決定
- VLM の `L/R` 二択は、上記が曖昧なときだけ fallback として使う構造へ変更
