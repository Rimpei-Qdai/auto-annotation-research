# 2026-04-07 軌道オーバーレイ Prompt 調査メモ

## 1. 目的

各フレームに 3 秒先の予測軌道を描画している現行入力に対して、VLM がその描画を背景の一部ではなく「将来運動の明示的な cue」として読むように、既存研究を踏まえた prompt 設計指針を整理する。

---

## 2. 参照した研究

### 2.1 Motion Prompting: Controlling Video Generation with Motion Trajectories

- 出典:
  - CVPR 2025
  - https://openaccess.thecvf.com/content/CVPR2025/html/Geng_Motion_Prompting_Controlling_Video_Generation_with_Motion_Trajectories_CVPR_2025_paper.html
- 要点:
  - 動的な運動はテキストだけでは表現しづらく、trajectory のような motion prompt を別チャネルとして与える方が有効。
  - trajectory は object motion / global scene motion / sparse temporal motion を柔軟に表せる。
- 本件への示唆:
  - 赤い軌道は「画像上の装飾」ではなく、将来運動を表す motion prompt として明示的に説明すべき。
  - prompt の先頭で「まず赤い軌道だけを読む」手順を入れるのが自然。

### 2.2 Adaptive Keyframe Sampling for Long Video Understanding

- 出典:
  - CVPR 2025
  - https://openaccess.thecvf.com/content/CVPR2025/html/Tang_Adaptive_Keyframe_Sampling_for_Long_Video_Understanding_CVPR_2025_paper.html
- 要点:
  - Video MLLM では、入力の一部しか使えないため、prompt relevance と coverage を意識した情報選択が重要。
- 本件への示唆:
  - 4 フレームしか見せない以上、prompt は各フレームから何を見るべきかを強く指定した方がよい。
  - 「全フレームで一貫する解釈を選ぶ」という指示を明示する。

### 2.3 On the Consistency of Video Large Language Models in Temporal Comprehension

- 出典:
  - CVPR 2025
  - https://openaccess.thecvf.com/content/CVPR2025/html/Jung_On_the_Consistency_of_Video_Large_Language_Models_in_Temporal_CVPR_2025_paper.html
- 要点:
  - Video-LLM は時間的一貫性に弱く、質問や設定に敏感。
  - 単純な prompting 改善だけでは不安定で、整合性確認が重要。
- 本件への示唆:
  - 最終分類 prompt に「4 フレーム全体で整合しているか確認する」手順を入れる。
  - 交差点・lane change・turn の分岐で、軌道 cue と背景 cue の矛盾をチェックさせる。

### 2.4 PoseTraj / trajectory-guided video control 系

- 出典:
  - CVPR 2025
  - https://openaccess.thecvf.com/content/CVPR2025/html/Ji_PoseTraj_Pose-Aware_Trajectory_Control_in_Video_Diffusion_CVPR_2025_paper.html
- 要点:
  - trajectory は 3D pose や wide-range motion を制御するための明示的条件として扱うと有効。
- 本件への示唆:
  - trajectory を scene appearance と分離した condition として扱う設計は妥当。
  - turn と lane-change を同じ「左移動」ではなく、異なる motion cue として分けて表現するのが望ましい。

---

## 3. Prompt 設計指針

今回の調査から、軌道オーバーレイ付き動画の prompt は次の方針が適切だと判断した。

1. 赤い軌道を motion prompt と明示する
2. まず軌道だけを読ませ、その後に背景文脈を見る
3. 軌道 cue を turn / lane-change / straight に離散化させる
4. 4 フレーム全体で一貫する解釈を求める
5. 背景の印象より軌道 cue を優先する

特に重要なのは、

- 「まず軌道 alone」
- 「次に緑線との相対関係」
- 「最後に交差点・道路形状」

という順番を prompt に書くこと。

## 4. 実装に反映した変更

### 4.1 Level 1

- `trajectory_motion_cue` を新設
- `STRAIGHT_CUE`
- `LEFT_TURN_CUE`
- `RIGHT_TURN_CUE`
- `LEFT_LANE_CHANGE_CUE`
- `RIGHT_LANE_CHANGE_CUE`
- `AMBIGUOUS`

これにより、赤い軌道の解釈が reasoning 文の中に埋もれず、Graph 側に直接渡せるようにした。

### 4.2 Level 2

- Level 1 の `trajectory_motion_cue` を明示的に参照
- 軌道 overlay の読み方を再掲
- センサと軌道 cue の整合を見るように指示

### 4.3 Level 3

- `trajectory_motion_cue` を最上位の判断材料として追加
- 「赤い軌道が連続的に曲がるなら TURN 優先」
- 「横移動主体で交差点が弱いなら LANE CHANGE 優先」
- 「背景だけで曲がって見えても、赤い軌道が直進なら TURN にしない」

### 4.4 Graph

- `trajectory_motion_cue` を Graph concept に追加
- turn / lane change の支持スコアに直接反映

---

## 5. 期待効果

- 赤い軌道が prompt 上で主役になる
- 左折 / 右折 / 車線変更の分離がしやすくなる
- 背景だけで turn を誤認する失敗を減らせる
- trajectory overlay を「見せているだけ」の状態から一歩進められる

---

## 6. 残課題

今回の変更は、まだ VLM が軌道を読むことに依存している。
理想的には次段で以下も入れたい。

- 3 秒軌道から curvature を直接計算
- green line に対する lateral offset を直接計算
- trajectory cue を VLM ではなく幾何特徴として併用

つまり、今回の変更は「trajectory を読むための prompt 改善」であり、次は「trajectory から特徴を直接計算する段階」に進むのが自然である。
