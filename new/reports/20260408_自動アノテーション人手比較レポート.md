## 6. ログから見える今回の失敗要因

## 6.1 `trajectory_motion_cue` が 99/99 件で `LEFT_TURN_CUE`

最新 run の `inference_process.jsonl` から、各 sample の最新 completed 結果を集計すると以下だった。

- `trajectory_motion_cue = LEFT_TURN_CUE`: 99 / 99
- `trajectory_relation = PARALLEL`: 99 / 99
- `intersection_detected = NO`: 99 / 99
- `direction_change = STRAIGHT`: 99 / 99
- `visual_shift = NO_SHIFT`: 99 / 99

これは極めて重要である。

今回の prompt 改善で導入した `trajectory_motion_cue` は、「赤い 3 秒軌道を先に読む」ことを目的としていたが、実際には

- 他の幾何項目は全部 `直進・平行・交差点なし`
- なのに `trajectory_motion_cue` だけが全部 `LEFT_TURN_CUE`

という内部矛盾を起こしている。

このため、Graph は `trajectory_motion_cue` を最も強い turn evidence として解釈し、左折候補を大量に押し上げた。

### 代表例

- sample 99
  - 人手: 等速走行
  - 自動: 左折
  - `trajectory_motion_cue = LEFT_TURN_CUE`
  - しかし Level 1 の他項目は `PARALLEL / NO / STRAIGHT / NO_SHIFT`
- sample 100
  - 人手: 等速走行
  - 自動: 減速
  - `trajectory_motion_cue = LEFT_TURN_CUE`
  - Graph 上位候補の 1 位は左折、2 位が減速

sample 100 では最終的に減速へ落ちたが、sample 99 ではそのまま左折に吸われている。

## 7. 改善案

## 優先度 A: まず直すべきもの

### A1. `trajectory_motion_cue` をそのまま信用しない

今回最大の原因である。

改善案:

- `trajectory_motion_cue` を Graph の強い evidence にする前に整合チェックを入れる
- `PARALLEL / NO / STRAIGHT / NO_SHIFT` と矛盾する場合、cue の重みを大きく下げる
- cue を単一値ではなく `support` と `confidence` に分ける
- cue が全件同一値に崩れていないかを run 単位で監視する

最低限、今回のように

- `LEFT_TURN_CUE`
- なのに `direction_change = STRAIGHT`

という場合は、その cue をそのまま turn に使わない方がよい。

### A2. `speed_change_rate` は clip 内の連続センサ窓で再計算する

今回の `timestamp_diff_sec` は中央値 14503.99 秒であり、判定窓として不適切である。

改善案:

- taxi 全体の前サンプルとの差分ではなく、当該 clip 近傍 1～3 秒のセンサ列から計算する
- その窓が取れないなら、`speed_diff` 系特徴を弱い evidence に落とす
- `delta_t > 10s` などのときは時間正規化特徴を無効化する

### A3. `trajectory_motion_cue` と幾何特徴の矛盾を explicit に持つ

改善案:

- `trajectory_motion_cue = LEFT_TURN_CUE`
  - かつ `trajectory_relation = PARALLEL`
  - かつ `intersection = NO`
  - かつ `direction_change = STRAIGHT`

のときは `contradiction edge` を作る

今は cue が turn candidate を一方的に押し上げるだけなので、矛盾側も Graph に持たせるべきである。

## 優先度 B: 次にやるべきもの

### B1. Level 1 を deterministic 幾何へ戻す

前回レポートでも主張した通り、本筋はここである。

改善案:

- projected trajectory の横変位から `lane_crossing` を算出
- trajectory 曲率から `turn_direction` を算出
- `trajectory_motion_cue` は VLM 出力ではなく、まず幾何特徴から組み立てる

### B2. 左折と車線変更(左)の構造差を明示する

今回は左方向 cue がすべて左折へ吸われた。

改善案:

- `intersection = YES` のときだけ左折を強く押す
- `intersection = NO` かつ lateral shift 優勢なら車線変更(左)を押す
- `LEFT_TURN_CUE` と `LEFT_LANE_CHANGE_CUE` の境界を prompt と Graph の両方で再定義する

## 8. まとめ

今回の run は、カバレッジは 99% と高かったが、一致率は 19.19% まで低下した。

悪化の中心は、

- `trajectory_motion_cue` が 99/99 件で `LEFT_TURN_CUE`
- それが Graph と Level 3 に流れ、左折 47 件という偏りを作った

ことである。

一方で、減速や停止の扱いは一部改善しており、

- speed-state の矛盾解消
- stop bias の抑制

は一定の効果が見られた。

したがって次の優先課題は、

- `trajectory_motion_cue` の重み付けと整合チェックの修正
- `speed_change_rate` の計算窓の見直し
- Level 1 幾何特徴の deterministic 化

である。
