# L2M/CoT + Graph 方式の失敗要因詳細調査レポート

**作成日**: 2026年4月13日  
**対象プロジェクト**: タクシー動画の運転行動自動アノテーション  
**対象期間**: 2026年3月25日 〜 2026年4月13日  
**目的**: 今回の L2M/CoT + Graph 方式が安定して精度向上しなかった理由を、設計・実装・評価の観点から詳細に整理する

---

## 1. 結論

今回のやり方がうまくいかなかった主因は、**推論を深くしたこと自体**ではなく、

- 強かった Ver.4 の直接視覚判断を
- 品質の低い中間表現へ圧縮し
- その中間表現を自然言語で何度も再解釈する構造にした結果、
- 誤りが段階的に増幅された

ことにある。

より端的に言えば、

**「理由を考えさせる」前に、「考える材料」を壊してしまった**

のが本質である。

そのため、

- L2M/CoT
- Graph
- cue validation
- 後処理ルール

を個別に改善しても、全体は 40% 前後から抜け出せず、run によっては 19% 台まで崩れた。

## 5. 失敗の構造

今回の失敗は、単一原因ではなく、次の 4 層が重なっていた。

1. **設計レベルの失敗**
2. **中間表現レベルの失敗**
3. **最終判定レベルの失敗**
4. **評価運用レベルの失敗**

以下で順に整理する。

---

## 6. 設計レベルの失敗

### 6.1 Ver.4 の強みを主経路として残さなかった

Ver.4 の強みは、

- 赤軌道をそのまま見せる
- 4フレームの時系列をそのまま見せる
- 直接分類させる

という、非常にシンプルな構造にあった。

一方、L2M/CoT 導入後は、

```text
画像
↓
Level 1 要約
↓
Level 2 要約
↓
Graph summary
↓
Level 3 再判定
```

のように、**直接視覚判断を主経路から外してしまった**。

その結果、

- 画像そのものの強い情報

よりも、

- 中間の文章要約
- 中間の離散ラベル

が最終判断を支配する構造になった。

これは設計上の大きな転換だったが、その効果を baseline と同条件で検証しないまま進めてしまった。

### 6.2 reasoning を「補助」ではなく「本流」にしてしまった

本来、L2M/CoT や Graph が効きやすいのは、

- 停止 vs 減速
- 左右折 vs 車線変更
- 等速 vs 加減速

のような境界事例である。

しかし今回は、

- 全件に reasoning をかける
- すべての sample を Level 1/2/3 に通す

構成にしたため、

- もともと簡単だった sample まで中間表現に変換され
- その変換誤りの影響を受ける

構造になった。

つまり、

- 難しい問題だけを助ける仕組み

ではなく、

- 全 sample を不安定なパイプラインに通す仕組み

になってしまった。

### 6.3 Graph が「真の graph reasoning」になっていなかった

Graph は研究上の発想としては有望だったが、実装上は主に

- 概念列挙
- score 付け
- summary 生成

に留まっていた。

つまり、実態は

- 深い graph inference

ではなく、

- 関係付き特徴列挙

に近かった。

そのため、Graph を入れても

- 強い contradiction resolution
- 多段の関係伝播
- label node への構造的制約

が十分に働かず、最終段の自然言語判断へ依存したままだった。

## 7. 中間表現レベルの失敗

### 7.1 Level 1 の幾何特徴が長期間ほぼ崩壊していた

2026-04-07 レポートでは、成功 96 件に対して

- `trajectory_relation = PARALLEL`: 96 / 96
- `intersection_detected = NO`: 96 / 96
- `direction_change = STRAIGHT`: 96 / 96
- `visual_shift = NO_SHIFT`: 96 / 96

だった。

これは、Level 1 が実質的に

- 直進
- 交差点なし
- 車線平行
- シフトなし

しか返せていないことを意味する。

この状態で L2M/CoT を回しても、

- 交差点依存の左右折
- lane crossing に依存する車線変更

はほぼ分離できない。

つまり、**最初の入力特徴が死んでいた**。

### 7.2 Level 2 の速度要約がセンサ実態と矛盾していた

2026-04-07 レポートでは、

- `acceleration_cause = DRIVER_ACCEL`: 88 / 96
- `speed_trend = STABLE`: 96 / 96
- `speed_diff < -0.5` なのに `DRIVER_ACCEL`: 31 件
- `brake = 1` なのに `DRIVER_ACCEL`: 27 件

と整理されている。

つまり、

- ブレーキが入っている
- 速度差分がマイナス

にもかかわらず、

- Level 2 が「加速由来」「安定」

と要約してしまっていた。

その結果、

- 等速走行が加速 / 減速へ崩れる
- 減速が停止へ吸われる
- 左折しながら加速のような矛盾候補が立つ

状態になった。

### 7.3 `trajectory_motion_cue` は有望だったが、最初の実装で一方向に崩壊した

2026-04-08 run では、

- `trajectory_motion_cue = LEFT_TURN_CUE`: 99 / 99

になり、他の幾何項目が

- `PARALLEL / NO / STRAIGHT / NO_SHIFT`

であるにもかかわらず、cue だけが左折へ一極化した。

その結果、

- 左折 47 件
- 一致率 19.19%

まで崩れた。

ここで重要なのは、

- 新特徴を入れたこと自体が悪かったのではなく
- **validation 前に Graph の強い evidence として使ってしまった**

ことが失敗だった点である。

### 7.4 deterministic feature を入れても、今度は stop / turn の過剰寄せが出た

赤軌道から直接

- 曲がり方向
- 曲率
- 横偏位
- 線の長さ
- point-like 性

を取る改善は前進だった。

しかし、その後の run では

- `trajectory_point_like=True`
- `trajectory_visual_speed_state=STOPPED`
- `stop_likelihood=HIGH`

が stop を過剰に押し、

- `intersection_detected=YES/UNCERTAIN`
- 左右 turn cue

が turn を過剰に押した。

つまり、

- 特徴抽出を deterministic にしただけでは不十分で
- その feature をどう使うかの制約設計が弱かった

ことが分かった。

## 8. 最終判定レベルの失敗

### 8.1 Level 3 が Graph 上位候補を覆してしまう

複数のレポートで、

- Graph 上では speed-event が強い
- しかし Level 3 が最終的に turn を返す

という事例が確認されている。

具体例として、2026-04-13 最新 run 改善対応後分析では、

- sample 23
- sample 85

などで、Graph 上は `加速` / `減速` が上位でも Level 3 が `右折` を返したと整理されている。

これは、

- Graph が候補順位を作っても
- Level 3 が自由生成に近い形で再解釈してしまう

ためである。

つまり、**Graph を入れても最終段に拘束力がなかった**。

### 8.2 Level 3 が画像より文章要約に引っ張られる

L2M/CoT 導入後の Level 3 は、

- Level 1 の要約
- Level 2 の要約
- Graph summary
- センサーヒント
- ルール文

をまとめて読んで最終判断していた。

この構造では、VLM が

- 画像から直接判断する

のではなく、

- 他段が作った文章を再解釈する

仕事に寄ってしまう。

その結果、

- 一つの summary が崩れる
- ルール文が強すぎる
- cue が極端に寄る

といった局所問題が、最終判定を大きく壊す。

### 8.3 出力構造が自由すぎて、誤パースや説明崩れも混ざった

過去には

- `category=0` への収束
- JSON パース失敗
- 冗長・反復的な reasoning

も発生している。

これらは最終的に一部修正されたが、そもそも

- 自由生成テキストからラベルを引く

構造自体が不安定だった。

これは、分類器というより

- 説明文生成器の出力を後からラベル化している

ことに近い。

## 9. 評価運用レベルの失敗

### 9.1 baseline と同条件で比較せずに改善を積み重ねた

Ver.4 の 51.52% は以前から分かっていたが、その後の reasoning 系改善では

- 同一 100 サンプル
- 同一条件
- 同一評価指標

で Ver.4 を再現しないまま、L2M/CoT 系を伸ばそうとしていた。

そのため、

- 本当に改善したのか
- baseline を壊しただけなのか

の切り分けが遅れた。

### 9.2 sample ごとの「改善 / 悪化」管理が弱かった

これまでの評価は主に

- Accuracy
- Macro F1
- クラス別傾向

で見ていた。

しかし本当に必要だったのは、

- baseline で正解だったのに reasoning で悪化した sample
- baseline で誤りだったが reasoning で改善した sample

の明示的な追跡である。

これが十分でなかったため、

- 局所改善に見えた変更が
- 全体では regressions を大量に生んでいた

可能性を見抜きにくかった。

### 9.3 実行不具合が研究評価を汚した時期があった

2026-04-07 前後には、

- `cpu` と `cuda:0` の device mismatch
- `0` へのフォールバック

も発生していた。

これは後に修正されたが、少なくとも一時期は

- 推論設計の評価
- 実行基盤の不具合

が混ざった状態で結果を見ていた。

研究の設計評価としては、これは大きなノイズだった。

## 10. うまくいかなかった本質

以上をまとめると、本質的な失敗は次の 3 つに集約できる。

### 10.1 入力特徴の改善より、推論段の追加を優先してしまった

このタスクで実際に精度を押し上げたのは、

- 4フレーム化
- 赤軌道の Visual Prompting

であり、どちらも「入力特徴」の改善だった。

一方 L2M/CoT 以降は、

- 低品質な特徴
- 不安定な summary

を前提に reasoning を深くしたため、伸びなかった。

### 10.2 reasoning を主経路にしてしまい、強い baseline を壊した

本来は、

- baseline を主経路
- reasoning を補助経路

とすべきだった。

しかし今回は逆に、

- reasoning を本流
- baseline 的な直接判断を従属

にしてしまった。

そのため、もともと簡単に解けていた sample まで不安定化した。

### 10.3 Graph / CoT を「説明性の向上」と「精度向上」で同時に期待しすぎた

Graph や CoT は、

- 説明可能性
- 中間概念の整理

には有効だった。

しかし、今回の実装ではそのまま

- 精度向上

まで担わせようとした。

結果として、

- 説明用の中間表現
- 最終分類用の強拘束

の役割が混ざり、どちらも中途半端になった。

---

## 11. 今回の調査から導かれる判断

今回の調査から、以下を判断する。

### 11.1 継続すべきもの

- 赤軌道そのものを使う発想
- deterministic な幾何 feature 抽出
- Graph を ranking / contradiction 整理に使う発想
- reasoning 系コードを研究資産として残すこと

### 11.2 縮小・停止すべきもの

- 全件に対する L2M/CoT 常時適用
- 自由生成に近い Level 3 最終判定
- 中間要約を主経路にする設計
- ローカルルールの継ぎ足しで全体精度を上げようとする方針

### 11.3 次の方針

今後は、

- **Ver.4 baseline-first**
- **reasoning-as-corrector**

へ切り替えるのが妥当である。

つまり、

1. Ver.4 を現在の評価セットで再現する
2. baseline を主経路に戻す
3. L2M/CoT / Graph / Verifier は曖昧ケースだけに使う

という順序にする。

---

## 12. 最終まとめ

今回のやり方がうまくいかなかったのは、

- L2M/CoT が悪いから

ではない。

本当に問題だったのは、

- 強かった visual baseline を維持せず
- 不安定な中間表現へ変換し
- その変換結果に最終判定を依存させた

ことである。

したがって、今回の教訓は次の一文に集約できる。

**推論を深くする前に、baseline の強い視覚判断を壊さない設計にしなければならない。**
