# VLM 軌道描画認識調査メモ

## 1. 調査目的

Qwen3-VL を主軸にした recent run では、

- `summary_v3_prompt_split` が反映されていた
- しかし VLM 単体は `N -> R -> C` にほぼ崩壊した

ことが分かった。

このとき、原因が

- VLM の能力限界なのか
- 軌道描画自体の設計不備なのか

を切り分けるため、trajectory summary の描画仕様を再確認した。

---

## 2. 判明した根本原因

### 2.1 左右の符号が描画と prompt で逆転していた

コード確認の結果、summary 画像への座標投影では

- 車両座標系の `+Y = 左`

であるにもかかわらず、画像側では

- `+Y` を **右側** に描画していた

ことが分かった。

そのため、実際の見え方は

- 左回転 sample が画像右側へ曲がる
- 右回転 sample が画像左側へ曲がる

という状態になっていた。

一方で prompt 側では一貫して

- 画像左側に曲がれば `B`
- 画像右側に曲がれば `C`

という前提で説明していた。

つまり、**VLM に与えていた図の左右と、prompt が説明していた左右が矛盾していた**。

これは VLM の認識問題というより、まず入力設計の不整合である。

---

## 3. 修正内容

### 3.1 좌右符号の修正

`_trajectory_to_canvas_points()` を修正し、

- `+Y = 左` が画像左側に
- `-Y = 右` が画像右側に

出るように反転した。

### 3.2 左右アンカーの追加

VLM が synthetic summary を読みやすいよう、画像5・6に

- `LEFT`
- `RIGHT`

の明示アンカーと色付きサイド帯を追加した。

これは中間判断ラベルではなく、**座標系の向きそのものを固定するための参照情報**である。

### 3.3 終点方向の明示

終点には従来の黄色マーカーに加えて、**進行方向を示す arrowhead** を追加した。

これにより、

- 軌道がどちら側に終わるか
- 最終的な進行向きがどちらか

を VLM がより取りやすくなることを狙った。

### 3.4 prompt との整合

prompt 側には最小限の変更として、

- 画像5・6 の `LEFT / RIGHT` アンカーは車両基準の左右である

ことだけを追加した。

今回は prompt を複雑化するのではなく、**描画と説明の左右対応を一致させること**を目的とした。

---

## 4. 今回の修正の意味

今回の修正は「VLM にもっと賢く考えさせる」ためのものではない。

むしろ、

- VLM に見せている図の意味が prompt と一致しているか
- 左右の幾何が画像として正しく表現されているか

という、**視覚入力の整合性を確保する修正**である。

そのため、この修正後に精度が改善した場合、それは

- prompt trick が効いた

というより、

- **VLM に与える軌道描画が初めて正しい左右意味を持った**

と解釈するのが妥当である。

---

## 5. 今後の確認ポイント

次回 run では特に以下を見る必要がある。

1. `stage3_turn_choice` で `C` 一辺倒が崩れるか
2. `A -> C` の誤りが減るか
3. `B` と `C` の raw 出力分布が左右でバランスするか
4. graph rescue なしでも VLM raw 出力が改善するか

---

## 6. まとめ

今回の調査で判明した最大の問題は、

**trajectory summary の左右描画が prompt の左右説明と逆だったこと**である。

したがって、VLM が軌道描画を適切に認識できていなかった原因の少なくとも一部は、

- モデル能力そのもの

ではなく、

- **入力図の左右意味が不整合だったこと**

にある。

今回の修正は、その不整合を解消し、VLM が左右の幾何を読める前提条件を整えるものである。
