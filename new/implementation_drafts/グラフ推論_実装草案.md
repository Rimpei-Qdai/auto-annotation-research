# グラフ推論案 実装草案

## 1. 目的

現状の自動アノテーションは、`Qwen2-VL-7B + 軌道描画 + L2M/CoT` まで進んでいる一方で、出力が `等速走行(1)` と `停止(4)` に大きく偏っている。  
本案では、最終ラベルを直接予測するのではなく、途中で `運転行動を構成する中間概念` を明示的に出力し、その関係から最終ラベルを決めることで、右左折・車線変更・加減速の識別を改善する。

---

## 2. 基本アイデア

L2M/CoT の 3 段階推論を維持しつつ、各段階の役割を以下のように再定義する。

- Level 1: 映像・軌道・センサから中間概念を抽出する
- Level 2: 抽出された中間概念の整合性を検証する
- Level 3: 整合性の取れた概念集合から最終ラベルを決定する

ここでいう中間概念は、以下のようなものを想定する。

- `speed_state`: stop / constant / accelerating / decelerating
- `trajectory_direction`: straight / left / right
- `trajectory_curvature`: low / medium / high
- `intersection_state`: yes / no / unknown
- `lane_crossing_state`: yes / no / unknown
- `turn_signal_state`: left / right / off
- `stop_likelihood`: high / medium / low
- `turn_likelihood`: left / right / none

この中間概念群をグラフとみなし、概念同士の `supports / contradicts / consistent_with` の関係を明示的に扱う。

---

## 3. 今のプロジェクトでやる意味

現状の問題は、最終ラベルの直接推論に失敗しているというより、以下のような `途中判断の曖昧さ` がそのまま最終出力に流れていることにある。

- 左に動いているが、左折なのか車線変更なのかが曖昧
- ブレーキ ON だが、減速なのか停止なのかが曖昧
- 軌道は曲がっているが、右折なのかただの進路補正なのかが曖昧

このため、「ラベルを当てる」より前に、「何を見てそう判断したか」を部品に分ける方が改善余地が大きい。

## 5. 実装のイメージ

### Step 1: 中間概念を定義する

まずは少数の概念に絞る。

- `speed_state`
- `trajectory_direction`
- `trajectory_curvature`
- `intersection_state`
- `lane_crossing_state`
- `turn_signal_state`

最初から概念を増やしすぎると不安定になるので、現状の課題に直結するものだけにする。

### Step 2: 既存の Level 1 を概念抽出フェーズに変える

今の Level 1 は `road_shape` や `trajectory_relation` を返している。  
これを拡張して、以下のような JSON を返すようにする。

```json
{
  "concepts": {
    "speed_state": "DECELERATING",
    "trajectory_direction": "RIGHT",
    "trajectory_curvature": "MEDIUM",
    "intersection_state": "YES",
    "lane_crossing_state": "NO",
    "turn_signal_state": "RIGHT"
  }
}
```

### Step 3: Level 2 で概念の整合性を検証する

例:

- `speed_state=DECELERATING` と `brake=ON` は整合
- `trajectory_direction=RIGHT` と `turn_signal_state=LEFT` は矛盾
- `intersection_state=NO` と `lane_crossing_state=YES` は車線変更寄り

この段階では、最終ラベルをまだ決めず、「どの概念がどのラベルを支持しているか」を整理する。

### Step 4: Level 3 で概念グラフからラベルを選ぶ

例:

- `trajectory_direction=RIGHT`
- `intersection_state=YES`
- `turn_signal_state=RIGHT`

なら `右折` を優先する。

- `trajectory_direction=LEFT`
- `intersection_state=NO`
- `lane_crossing_state=YES`

なら `車線変更(左)` を優先する。

## 6. データ構造案

`driving_graph.py` では、以下のような構造で持つ。

```python
{
  "nodes": {
    "speed_state": "DECELERATING",
    "trajectory_direction": "RIGHT",
    "intersection_state": "YES"
  },
  "edges": [
    {"source": "trajectory_direction", "target": "turn_signal_state", "type": "consistent_with"},
    {"source": "intersection_state", "target": "lane_crossing_state", "type": "contradicts"}
  ],
  "label_support": {
    "7": 0.82,
    "9": 0.15
  }
}
```

## 7. 実装順序

### Phase 1

- `driving_graph.py` を追加
- `speed_state / trajectory_direction / turn_signal_state` の 3 概念だけ実装
- `heron_l2m_pipeline.py` に概念 JSON を返す口を追加

### Phase 2

- `intersection_state / lane_crossing_state / trajectory_curvature` を追加
- Level 2 の整合性ルールを拡張
- 推論ログに概念グラフを保存

### Phase 3

- 最終ラベル選択を概念グラフ主導に寄せる
- バッチ推論結果を既存結果と比較

## 8. 評価方法

### 主評価

- 全体 Accuracy
- Macro F1
- クラス別 Recall

### 重点観察クラス

- 3: 減速
- 6: 左折
- 7: 右折
- 8: 車線変更(左)
- 9: 車線変更(右)

### 特に見たい誤認識

- `左折 ↔ 車線変更`
- `右折 ↔ 等速走行`
- `減速 ↔ 停止`
- `減速 ↔ 等速走行`

---

## 9. 期待効果

- `1` と `4` への過剰集中を緩和できる可能性がある
- 少数クラスを「直接当てる」のではなく「概念の組み合わせ」で拾いやすくなる
- 論文上の新規性として、`構造化中間概念を介した運転行動推論` を主張しやすい

---

## 10. リスク

- 中間概念自体の抽出精度が低いと逆効果になる
- 概念を増やしすぎると推論が不安定になる
- 現状の VLM が概念抽出に失敗すると、ラベル精度も落ちる

---

## 11. 撤退条件

以下が見えた場合は、Graph 主軸から RAG 補助案へ切り替える。

- 概念抽出の安定性が低く、同一クラス内でも一貫しない
- `右折 / 車線変更 / 減速` の Recall が改善しない
- `1` と `4` の偏りがほぼ解消しない

---

## 12. この案を本命にする理由

RAG より実装は重いが、研究の芯としては最も強い。  
今のプロジェクトにすでにある `speed_diff`, `gyro_z`, `blinker`, `trajectory visualization`, `L2M 3段階構造` をそのまま活かしやすく、論文では「単なる CoT ではなく、構造化概念推論を行った」と言える。
